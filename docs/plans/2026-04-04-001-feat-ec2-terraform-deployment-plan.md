# feat: EC2 Terraform deployment with Nginx + Supervisor

## Overview

Add a `terraform/` directory containing all configuration needed to provision a fresh EC2
instance and fully deploy the Santo Domingo House Price API — Nginx on port 80, Uvicorn
managed by Supervisor on port 8000, model and data uploaded from the local machine — in a
single `terraform apply` with no manual steps.

## Problem Frame

The app currently requires manual SSH setup to deploy. Every new instance means hand-running
apt installs, git clone, pip install, and service configuration from memory. A reproducible
Terraform config turns deployment into a one-command operation and eliminates drift between
instances. (see origin: docs/brainstorms/2026-04-04-ec2-terraform-deploy-requirements.md)

## Requirements Trace

- R1. EC2 `t3.small`, Ubuntu 22.04 LTS, AMI resolved via data source
- R2. Elastic IP attached for stable public address
- R3. SSH key pair generated by Terraform, saved to `terraform/key.pem` (chmod 600)
- R4. Security group: port 22 (CIDR variable), port 80 (open), all egress
- R5. EIP output after apply
- R6. All configurable values as Terraform variables; `terraform.tfvars.example` provided
- R7–R10. remote-exec #1: system packages, git clone, venv + pip install, mkdir -p for data/ and ml/
- R11–R12. File provisioners: upload `data/listings.csv` and `ml/model.pkl` from local machine
- R13. remote-exec #2: Supervisor config installed from cloned repo, log dir created, service started
- R14. remote-exec #2: Nginx config written, default site removed, new site enabled, Nginx reloaded
- R15. remote-exec #2: Health check retry loop exits non-zero if app does not become healthy
- R16. All Terraform files in `terraform/`; `key.pem` and `.terraform/` added to `.gitignore`

## Scope Boundaries

- No HTTPS/TLS — Nginx listens on port 80 only
- No dedicated VPC — uses the default VPC present in every AWS account
- No auto-scaling, load balancer, or multi-instance setup
- No RDS or external storage — model and data live on the instance
- No CI/CD — deployment is triggered manually via `terraform apply`
- No remote state backend — `terraform.tfstate` lives locally
- No imported key pairs — key pair generated fresh on each apply

### External References

- Ubuntu 22.04 cloud-init runs apt in background at boot — `cloud-init status --wait` required before any `apt-get` call to prevent lock conflicts

## Key Technical Decisions

- **Default VPC over dedicated VPC**: Default VPC exists in every AWS account and requires zero setup. A dedicated VPC adds ~6 resources (VPC, subnet, IGW, route table, route table association) for no benefit on a solo project.
- **All provisioners on `aws_instance` in declaration order**: Terraform executes provisioners on a resource in the order they are declared. This gives us the required sequence (remote-exec #1 → file provisioners → remote-exec #2) without null_resource scaffolding. Simpler and fewer moving parts.
- **`tls_private_key` (RSA 4096) over manual key import**: Key is generated alongside infrastructure, tracked in state, and destroyed with `terraform destroy`. Eliminates manual key management. `local_sensitive_file` with `file_permission = "0600"` ensures SSH accepts it.
- **`aws_eip` with `instance` argument**: Single resource, cleaner lifecycle than `aws_eip_association`. Works correctly with default VPC (no `depends_on` IGW needed since default VPC's IGW is pre-existing and not managed by Terraform).
- **`domain = "vpc"` on `aws_eip`**: Required for all non-EC2-Classic Elastic IPs in AWS provider 5.x. Omitting it produces a deprecation warning and will break in future provider versions.
- **cloud-init wait as first remote-exec step**: Ubuntu 22.04 AMIs run `unattended-upgrades` and cloud-init apt operations at boot. Without `cloud-init status --wait`, apt-get in remote-exec may hit the dpkg/apt lock and fail with "Could not get lock /var/lib/dpkg/lock".
- **`cp` not file provisioner for supervisor.conf**: The file is already on the server from `git clone`. Using `cp` in remote-exec #2 avoids an unnecessary network upload and preserves the source-of-truth relationship between the repo file and what's deployed.
- **`unlink` to remove Nginx default site**: Ubuntu's Nginx default site is a symlink in `sites-enabled/`. Unlinking it prevents port 80 conflicts between the default config and the new proxy config.
- **SSH CIDR defaults to `0.0.0.0/0`**: Convenient for solo use; lockable to a specific CIDR via `var.ssh_cidr`. Note: leaving this open is acceptable for development but should be tightened in any long-lived environment.

## Open Questions

### Resolved During Planning

- **scikit-learn==1.7.2 on PyPI?**: Confirmed present (`pip index versions scikit-learn` shows 1.7.2 as an available version).
- **AMI name filter**: `ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*` with owner `099720109477` (Canonical's canonical owner ID).
- **Venv path matches supervisor.conf?**: Yes — `deploy/supervisor.conf` hardcodes `/home/ubuntu/Project1/venv/bin/uvicorn`. The virtualenv created at R9 uses that exact path.
- **Default VPC or dedicated VPC?**: Default VPC. No VPC resources needed in `main.tf`.
- **data/ directory after git clone**: NOT created by `git clone` (listings.csv is gitignored, no other tracked files in data/). `mkdir -p` is required in remote-exec #1 before the file provisioner runs.

### Deferred to Implementation

- **AWS region to use**: Implementer sets in `terraform.tfvars` or `AWS_DEFAULT_REGION` env var. Default in `variables.tf` should be `us-east-1`.
- **map_public_ip_on_launch**: In default VPC with default subnet, public IPs are assigned by default. If the account has a non-default subnet, set `associate_public_ip_address = true` on the instance.
- **Nginx worker_processes**: Default (`auto`) is fine for t3.small; no tuning needed.

## High-Level Technical Design

> *This illustrates the intended approach and is directional guidance for review, not implementation specification. The implementing agent should treat it as context, not code to reproduce.*

```
terraform/
├── main.tf               ← All resources + provisioners
├── variables.tf          ← Input variable declarations
├── outputs.tf            ← Elastic IP output
└── terraform.tfvars.example  ← Documents required overrides

Resources in main.tf (declaration order → creation/execution order):
  data "aws_ami"              ← resolves Ubuntu 22.04 AMI ID
  tls_private_key             ← generates RSA 4096 key
  aws_key_pair                ← registers public key with AWS
  local_sensitive_file        ← writes private key to terraform/key.pem
  aws_security_group          ← ports 22 (var.ssh_cidr) and 80 (open)
  aws_instance                ← t3.small, Ubuntu AMI, SG, key
    provisioner "remote-exec" #1  ← cloud-init wait, apt, git clone,
                                     venv, pip install, mkdir -p
    provisioner "file"            ← upload data/listings.csv
    provisioner "file"            ← upload ml/model.pkl
    provisioner "remote-exec" #2  ← supervisor config, log dir,
                                     nginx config, service start,
                                     health check retry loop
  aws_eip                     ← attaches to instance, domain="vpc"

  output "elastic_ip"         ← printed after apply
```

Provisioner execution sequence on `aws_instance` (declaration order = execution order):

```
remote-exec #1 steps (inline):
  1. cloud-init status --wait
  2. sudo apt-get update && sudo apt-get install -y python3 python3-pip python3-venv git supervisor nginx
  3. git clone https://github.com/Maxi-Heartnet/Project1.git /home/ubuntu/Project1
  4. python3 -m venv /home/ubuntu/Project1/venv
  5. /home/ubuntu/Project1/venv/bin/pip install -r /home/ubuntu/Project1/requirements.txt
  6. mkdir -p /home/ubuntu/Project1/data /home/ubuntu/Project1/ml

file provisioner #1:
  local:  data/listings.csv
  remote: /home/ubuntu/Project1/data/listings.csv

file provisioner #2:
  local:  ml/model.pkl
  remote: /home/ubuntu/Project1/ml/model.pkl

remote-exec #2 steps (inline):
  1. sudo mkdir -p /var/log/predict-api
  2. sudo cp /home/ubuntu/Project1/deploy/supervisor.conf \
        /etc/supervisor/conf.d/predict-api.conf
  3. sudo supervisorctl reread && supervisorctl update && supervisorctl start predict-api
  4. write Nginx config to /etc/nginx/sites-available/predict-api
     (proxy_pass http://127.0.0.1:8000, proxy_set_header Host / X-Real-IP)
  5. sudo unlink /etc/nginx/sites-enabled/default
  6. sudo ln -s /etc/nginx/sites-available/predict-api \
        /etc/nginx/sites-enabled/predict-api
  7. sudo nginx -t && sudo systemctl reload nginx
  8. for i in 1..10: curl -sf http://localhost:8000/health && exit 0 || sleep 3
     exit 1 if loop exhausts (surfaces failure to Terraform)
```

- [ ] **Unit 2: Write complete main.tf with all resources and provisioners**

**Goal:** Implement all infrastructure resources and the full three-phase provisioner sequence in `main.tf`.

**Requirements:** R1, R2, R3, R4, R5, R7–R15

**Dependencies:** Unit 1

**Files:**
- Modify: `terraform/main.tf`

**Approach:**

*Data source:*
- `data "aws_ami" "ubuntu"` with filter `name = ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*`, `owners = ["099720109477"]`, `most_recent = true`

*Key pair:*
- `tls_private_key.main`: `algorithm = "RSA"`, `rsa_bits = 4096`
- `aws_key_pair.main`: `public_key = tls_private_key.main.public_key_openssh`
- `local_sensitive_file.ssh_key`: `content = tls_private_key.main.private_key_pem`, `filename = "${path.module}/key.pem"`, `file_permission = "0600"`

*Security group:*
- `aws_security_group.main`: ingress 22 from `var.ssh_cidr`, ingress 80 from `0.0.0.0/0`, egress all

*EC2 instance (`aws_instance.main`):*
- `ami = data.aws_ami.ubuntu.id`
- `instance_type = var.instance_type`
- `key_name = aws_key_pair.main.key_name`
- `vpc_security_group_ids = [aws_security_group.main.id]`
- Connection block: `type = "ssh"`, `user = "ubuntu"`, `private_key = tls_private_key.main.private_key_pem`, `host = self.public_ip`, `timeout = "5m"`
- Provisioner declaration order (executed in this order):
  1. `provisioner "remote-exec"` — cloud-init wait, apt, clone, venv, pip, mkdir
  2. `provisioner "file"` — listings.csv upload
  3. `provisioner "file"` — model.pkl upload
  4. `provisioner "remote-exec"` — supervisor, nginx, health check

*remote-exec #1 key steps:*
- `cloud-init status --wait` (resolves apt lock race)
- `sudo apt-get update -y && sudo apt-get install -y python3 python3-pip python3-venv git supervisor nginx`
- `git clone ${var.repo_url} /home/ubuntu/Project1`
- `python3 -m venv /home/ubuntu/Project1/venv`
- `/home/ubuntu/Project1/venv/bin/pip install -r /home/ubuntu/Project1/requirements.txt`
- `mkdir -p /home/ubuntu/Project1/data /home/ubuntu/Project1/ml`

*file provisioner #1:*
- `source = "${path.module}/../data/listings.csv"`, `destination = "/home/ubuntu/Project1/data/listings.csv"`

*file provisioner #2:*
- `source = "${path.module}/../ml/model.pkl"`, `destination = "/home/ubuntu/Project1/ml/model.pkl"`

*remote-exec #2 key steps:*
- `sudo mkdir -p /var/log/predict-api`
- `sudo cp /home/ubuntu/Project1/deploy/supervisor.conf /etc/supervisor/conf.d/predict-api.conf`
- `sudo supervisorctl reread && sudo supervisorctl update && sudo supervisorctl start predict-api`
- Write Nginx config via `sudo tee /etc/nginx/sites-available/predict-api` (heredoc in inline): `server { listen 80; location / { proxy_pass http://127.0.0.1:8000; proxy_set_header Host $$host; proxy_set_header X-Real-IP $$remote_addr; } }` — **use `$$host` and `$$remote_addr`** (double-dollar) so Terraform does not try to interpolate these as HCL variable references
- `sudo unlink /etc/nginx/sites-enabled/default || true`  ← `|| true` required: unlink fails if symlink already removed
- `sudo ln -sf /etc/nginx/sites-available/predict-api /etc/nginx/sites-enabled/predict-api`
- `sudo nginx -t && sudo systemctl reload nginx`
- Health check retry loop: `for i in $(seq 1 10); do curl -sf http://localhost:8000/health && exit 0; sleep 3; done; echo 'Health check failed'; exit 1`

*Elastic IP:*
- `aws_eip.main`: `instance = aws_instance.main.id`, `domain = "vpc"`, `depends_on = [aws_instance.main]`

**Patterns to follow:**
- `deploy/supervisor.conf` for exact paths the service expects
- `api.py` lifespan for understanding startup time (model load ~500ms; 10 × 3s = 30s retry window is ample)

**Test scenarios:**
- Happy path: `terraform apply` completes with exit 0; output shows elastic IP
- Happy path: `curl http://<EIP>/health` → `{"status":"ok","model_loaded":true}`
- Happy path: `curl http://<EIP>/` → HTTP 200 with HTML page title "Santo Domingo House Price Estimator"
- Happy path: POST to `http://<EIP>/predict` with valid JSON → HTTP 200 with `price_low`, `price_high`, `market_tier`
- Happy path: `terraform destroy` removes all resources with exit 0
- Error path: `terraform apply` when `data/listings.csv` is missing from local machine → file provisioner fails with a clear error before services are configured
- Error path: `terraform apply` when `ml/model.pkl` is missing from local machine → file provisioner fails with a clear error
- Edge case: applying twice (idempotence of infra resources) — aws_instance is not reprovisioned; only changed attributes trigger replacement

**Verification:**
- `terraform validate` passes
- `terraform plan` shows all expected resources with no errors
- After apply: EIP output is printed and the four endpoints are reachable

## System-Wide Impact

- **New directory**: `terraform/` at repo root — pure addition, no existing code affected
- **`.gitignore` change**: Adds Terraform exclusions; existing exclusions unchanged
- **Unchanged invariants**: `api.py`, `tests/`, `deploy/supervisor.conf`, `requirements.txt` — none modified. The Terraform config reads from these files but does not change them.
- **Deploy/supervisor.conf dependency**: Any future change to hardcoded paths in `supervisor.conf` (e.g., project path, venv path, log directory) must be mirrored in the Terraform provisioner scripts.
- **model.pkl / listings.csv not in repo**: The `terraform apply` operator must have both files locally. If a team member runs `terraform apply` without them, the file provisioners fail with a clear error (file not found on local machine) before any services are configured — no partial-deploy risk.

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| `cloud-init status --wait` not available | Present on all Ubuntu 22.04 AMIs (cloud-init is installed by default). If somehow absent, `apt-get` may fail with lock error on first run; retrying apply would fix it. |
| Provisioner ordering assumption | Terraform processes provisioners in declaration order on a single resource. If this behavior ever changes, the `data/` mkdir would race the file provisioner. Mitigation: the `mkdir -p` in remote-exec #1 and the directory existence guarantee from the clone are belt-and-suspenders. |
| `terraform.tfstate` lost | Manual AWS Console cleanup required (EC2, EIP, SG, key pair, tls key in state). Accepted per scope. Document recovery steps in `terraform/terraform.tfvars.example` comments. |
| SSH key stored in tfstate | `tls_private_key` stores the private key in state. Never commit `terraform.tfstate` to git (covered by .gitignore in Unit 3). This is standard practice for this pattern. |
| apt lock despite `cloud-init status --wait` | Some Ubuntu AMIs run `unattended-upgrades` outside cloud-init scope. Add `sudo systemctl stop unattended-upgrades || true` as a second apt-guard step after `cloud-init status --wait`. |
| Nginx `$host`/`$remote_addr` variables in Terraform heredoc | Dollar signs in Terraform `inline` strings are interpreted as variable references. Use `$$host` and `$$remote_addr` in the Nginx config string, which Terraform renders as literal `$`. |

## Documentation / Operational Notes

- `terraform.tfvars.example` should include usage instructions as comments: how to copy to `terraform.tfvars`, how to set `ssh_cidr` to `$(curl -s ifconfig.me)/32`, and a reminder that `data/listings.csv` and `ml/model.pkl` must be present locally before applying.
- After a successful deploy: `ssh -i terraform/key.pem ubuntu@<EIP>` to access the instance.
- To redeploy the app (e.g., after a code change): `terraform destroy && terraform apply` or SSH in and `git pull` + `sudo supervisorctl restart predict-api`.
- Log locations on the instance: `/var/log/predict-api/stdout.log` and `stderr.log`.
